### 9. 排查并修复了 warm-start run 的 checkpoint 误判

`Pilot B` 首次训练结束时，批量脚本日志报：

- `checkpoint 不存在: model_599.pt`
- `artifact verification failed`

彻底检查后确认：

- [Fact] 训练本身并没有失败，run 返回码为 `0`
- [Fact] 实际 run 目录中已经产出：
  - `model_1500.pt`
  - ...
  - `model_2098.pt`
- [Fact] stdout 中明确出现：
  - `Loading initial policy checkpoint from ... model_1499.pt`
  - `Learning iteration 1500/2099`

根因：

- [Inference + Evidence] 当前 `run_morl_train_sweep.py` 的旧核验逻辑写死假设最终 checkpoint 一定是 `model_{max_iterations-1}.pt`
- [Inference + Evidence] 这个假设只对冷启动成立，对 warm-start 不成立
- [Inference + Evidence] warm-start run 会沿用已加载 checkpoint 的 iteration 编号继续计数，因此 `Pilot B` 的最终 checkpoint 正确值应为 `model_2098.pt`，而不是 `model_599.pt`

修复：

- 在 `scripts/phase_morl/run_morl_train_sweep.py` 中新增 warm-start 期望 checkpoint 解析逻辑
- 现在会根据 `init_checkpoint=model_1499.pt` 和 `max_iterations=600` 推导出最终应检查 `model_2098.pt`
- 同时把 `RunResult.checkpoint_path` 的回写逻辑改为使用同一套解析结果

验证：

- 对真实 `Pilot B` run 目录做轻量核验时，修复后已能得到：
  - `expected=model_2098.pt`
  - `exists=True`
  - `verified=True`

   不再接受“训练通道都亮了，但固定命令实验里不走”的情况。

## 遇到的问题

### 1. `warm-start` 和 `resume` 不是一回事

最开始的风险点是：如果直接复用 `--resume --load_run --checkpoint`，很容易把“旧 run 断点续训”和“新 run 初始化加载 baseline 权重”混成一个语义。

当前处理：

- 显式新增 `--init_checkpoint`
- 明确禁止和 `--resume` 同时使用

但仍保留一个未决点：

- [Don't know] `runner.load()` 底层到底是只加载网络参数，还是还会带优化器状态

### 2. 旧脚本的根目录校验与当前工作区不一致

新 pilot runner 初版沿用了旧 sweep 的根目录判断，默认要求仓库根目录存在 `CLAUDE.md`。当前工作区实际上使用 `AGENTS.md`。

结果：

- [Fact] dry-run 首次失败并报“Invalid project root”
- [Fact] 已改为同时接受 `AGENTS.md` / `CLAUDE.md`

### 3. pytest tmpdir 在当前 Windows 环境下有权限噪声

表现：

- 使用 `tmp_path` 或 `--basetemp` 时，pytest 会在会话级临时目录清理阶段触发权限异常

当前处理：

- 把受影响测试改成仓库内自管临时目录
- 避开 pytest tmpdir 插件

结论：

- [Fact] 这是测试环境噪声，不是 Task 1 逻辑本身失败

### 4. warm-start run 的 checkpoint 编号不再从 0 开始

表现：

- 批量脚本首次把 `Pilot B` 误判为失败

本质：

- [Fact] warm-start 从 `model_1499.pt` 接着训练
- [Fact] 因此训练迭代从 `1500` 开始，最终 checkpoint 为 `model_2098.pt`
- [Inference + Evidence] 这不是训练失败，而是旧核验规则对 warm-start 不适用

### 5. 之前把“通道激活”错当成了“方案成立”

这个不是某一个 bug，而是在阶段三后半段默认接受了一个错误判据：

- 只要 `vel / energy / smooth / stable` 四个通道都能亮
- 只要不同权重策略之间能拉开差异
- 只要多种子稳定、Pareto 能算

我就倾向于把这条路线当成已经跑通了。

但现在回头看，这个判据不够严：

- 它没有强制要求“模型必须先会走”
- 也没有把 `S1` 固定命令前进能力当成每一轮修复后的硬验收项

所以这次不再把问题理解成“再加一点 pilot 就能补救”，而是明确承认：

- 旧路线本身在 reward 层级关系上就有问题
- 现在必须先重建训练架构，再继续后面的正式实验
