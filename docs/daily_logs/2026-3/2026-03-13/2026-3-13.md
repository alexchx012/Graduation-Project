### 3. 步骤 4.6b：组合验证配置确定

基于 `logs/eval/phase4_analysis/best_ppo_dr_analysis.json` 的训练曲线初判结果：

- 按“最终 100 iter 平均 reward 最高”口径，`best PPO = ppo_clip_low`，`best DR = dr_mass`
- 但 `ppo_clip_low` 的 3 种子方差明显偏大，且 `seed43` 曾出现一次 `std < 0` 数值不稳定崩溃后才重跑成功
- 因此组合验证实验不直接使用 `clip_low`，改为采用更保守的 `ppo_clip_high + dr_mass`

关键数据（last-100 iter mean reward）：

| 配置          | 3-seed mean ± std | 备注                                 |
| ------------- | ------------------ | ------------------------------------ |
| baseline      | 21.70 ± 6.58      | 对照组                               |
| ppo_clip_low  | 26.37 ± 13.88     | PPO 均值最高，但方差最大             |
| ppo_clip_high | 23.16 ± 6.46      | PPO 次优，但方差约为 clip_low 的 46% |
| dr_mass       | 42.53 ± 0.68      | DR 最优                              |

最终决定：

- 组合训练使用 `clip_param=0.3`
- DR 任务使用 `Isaac-Velocity-Rough-Unitree-Go1-ROS2Cmd-DRMass-v0`
- 通过 `python scripts/phase4-ppo-dr/run_phase4_train_sweep.py --mode combo` 自动执行 3 个种子（42/43/44）
- 已用 `--dry-run` 验证 3 条种子命令生成正确：`combo_best_seed42/43/44`

已同步更新 `scripts/phase4-ppo-dr/run_phase4_train_sweep.py` 中的 `COMBO_EXPERIMENTS` 默认配置，使其直接指向 `ppo_clip_high + dr_mass`。
此外修复了该脚本在 Windows 控制台下打印 `✓/⚠/✅/❌` 时触发的 GBK 编码崩溃，避免 combo 模式在前置检查阶段中断。
