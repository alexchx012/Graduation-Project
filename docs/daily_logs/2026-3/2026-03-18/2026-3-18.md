#### 确认的根因：指数核奖励 + 低权重 → 死梯度陷阱

**[Fact]** 训练日志显示 `track_lin_vel_xy_exp` 奖励在整个 1500 iter 训练过程中
始终为 `≈ 0.0045 ~ 0.0049`，从未上升。

**[Fact]** MORL 速度奖励函数 (`morl_rewards.py:31-45`)：

```python
def morl_track_vel_exp(env, command_name="base_velocity", scale=5.0):
    lin_vel_error = torch.sum(
        torch.square(
            env.command_manager.get_command(command_name)[:, :2]
            - asset.data.root_lin_vel_b[:, :2]
        ),
        dim=1,
    )
    return torch.exp(-scale * lin_vel_error)
```

**[Fact]** MORL 配置 (`morl_env_cfg.py:38-42`)：`weight=0.25`, `scale=5.0`

**[Inference + Evidence]** 数学推导：

| 配置                                             | 误差=1.0 时的奖励       | 每步有效信号                 |
| ------------------------------------------------ | ----------------------- | ---------------------------- |
| **MORL** (`scale=5.0, weight=0.25`)      | `exp(-5.0) ≈ 0.0067` | `0.25 × 0.0067 ≈ 0.0017` |
| **Baseline** (`1/std²=4.0, weight=1.5`) | `exp(-4.0) ≈ 0.018`  | `1.5 × 0.018 ≈ 0.027`    |

Baseline 信号强度约为 MORL 的 **16 倍**。

**[Inference + Evidence]** 因果链：

```
训练初期 policy 随机 → vx_meas ≈ 0
  → squared_error = (1.0 - 0)² = 1.0
  → reward = exp(-5.0) × 0.25 ≈ 0.0017
  → PPO advantage ≈ 0 → 无有效梯度
  → policy 收敛到 "站着不动"（局部最优）
  → 指数核在大误差区域梯度趋零 → 永远无法逃脱
  → 所有 20 个策略一致表现：J_speed ≈ 1.0
```

**[Inference]** 这是 RL 中经典的 **reward shaping plateau** 问题：
指数核奖励函数对精细跟踪有好的梯度特性（误差小时梯度大），
但当初始误差较大时形成"平坦高原"，PPO 几乎看不到梯度。
基线的 `weight=1.5` 足以突破初始阶段；MORL 的 `weight=0.25` 太弱，无法突破。

### 2. 修复方向评估与决策

识别三个候选修复方案，经分析选定方案 A：

| 方案                    | 描述                                                          | 预估效果                          | 决策                      |
| ----------------------- | ------------------------------------------------------------- | --------------------------------- | ------------------------- |
| **A: 降低 scale** | `scale=5.0 → 1.0`，`exp(-1.0)=0.368`，有效信号提升 54 倍 | 最直接，风险低                    | **✅ 选定**         |
| B: 替换奖励核           | 改用线性/Huber 核，大误差时仍有梯度                           | 彻底解决，但改变 reward landscape | ❌ 破坏与 baseline 可比性 |
| C: 课程学习             | 前期高速度权重(0.6)，逐步降到目标值(0.25)                     | 实现较复杂，需额外调度逻辑        | ❌ 与线性标量化理论冲突   |

**方案 A 选定理由**：

1. **信号强度验证**：修复后即使最弱权重 P2/P3/P4 (speed=0.1) 在初始误差时的有效信号
   `0.1 × exp(-1.0) = 0.037` 仍略高于 baseline `1.5 × exp(-4.0) = 0.027`
2. **函数形式不变**：仍为 `exp(-scale * error²)`，与 baseline 同族，论文无需额外辩护
3. **代码改动极小**：仅修改 `morl_env_cfg.py` 的 scale 参数值（1 行）
4. **不影响已有架构**：评估脚本、CLI weight override、训练 sweep 全部兼容

| 配置                                  | 误差=1.0 时有效信号 | vs Baseline        |
| ------------------------------------- | ------------------- | ------------------ |
| Baseline (1/std²=4.0, w=1.5)         | 0.027               | 参照               |
| MORL P1 旧 (speed=0.7, scale=5.0)     | 0.0047              | 0.17× ❌          |
| **MORL P1 新 (speed=0.7, scale=1.0)** | **0.258**     | **9.6×** ✅ |
| **MORL P4 新 (speed=0.1, scale=1.0)** | **0.037**     | **1.4×** ✅ |

**方案 B 排除理由**：换核后 MORL vs Baseline 变成苹果比橘子；且与其他 3 个目标
（均为指数核）不一致，引入理论不对称性。Phase 2 已证明提高权重即可解决速度跟踪
（Variant A: 1.5→3.5 显著改善），换核属于过度工程化。

**方案 C 排除理由**：MORL 线性标量化的核心是「固定权重 → 固定 trade-off 点 → Pareto 前沿」。
课程学习的动态权重破坏这一对应关系。且即使初始权重提高到 0.6，scale=5.0 下信号仍然不足。
