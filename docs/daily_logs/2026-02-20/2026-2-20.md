# 2026-02-20 工作日志

## 完成内容

### 1. 修复 Windows↔WSL2 ROS2 DDS 通信

定位并解决了 Go1 训练端（Windows/IsaacLab）无法接收 WSL2 ROS2 速度命令的问题。
根本原因是 WSL2 mirrored networking 下 loopback 地址不对称 + Fast-DDS 默认 multicast
无法跨 WSL2 边界。详细排查过程见 [2026-2-20-ros2-debug-detail.md](2026-2-20-ros2-debug-detail.md)。

修复涉及：

- `configs/ros2/fastrtps_win_to_wsl.xml` / `fastrtps_wsl_to_win.xml`：非对称地址策略，强制 UDPv4，禁用 multicast 元数据路径
- `scripts/go1-ros2-test/train.py`：补全 `FASTDDS_BUILTIN_TRANSPORTS`、`FASTDDS_DEFAULT_PROFILES_FILE` 等环境变量
- `scripts/go1-ros2-test/run/` 下两套 run 脚本：DDS 环境变量对齐、`wslpath` 修复、发布节点存活检查

## 遇到的问题

### WSL2 与 Windows DDS 通信不通

来自昨日日志，`cmd_zero_fallback_count` 持续增长，`cmd_vx` 始终为 0。
共经历 5 轮 debug 测试才定位根因，中途还遇到 FastRTPS XML 语法错误、
`wslpath` 路径转换失败、正式训练脚本 DDS 环境变量缺失等次要问题。

### cmd_timeout_count 中后期持续波动

训练全程 `cmd_timeout_count` 均值约 46，末段 100 轮均值约 43，未降至 0。
推测原因：WSL2 ROS2 发布频率与 Isaac 仿真步长之间存在异步抖动。
当前 `cmd_vx=1.0`、`cmd_zero_fallback_count` 末轮为 0，链路本身无问题，
但后续评估时需关注实际命令接收稳定性。

### `run_vf_go1_two_stage_training.ps1` 在 Stage-2 启动失败

在“Stage-1 + Stage-2 合并为同一 W&B run”改造后，首次复跑出现 Stage-2 失败。

- 首次报错：`ConfigError: Attempted to change value of key "log_dir"`
- 继续排查后报错：`ConfigError: Attempted to change value of key "runner_cfg"`
- 结论：不是 ROS2 通信中断，而是 W&B 对同一 run 的配置键改值限制导致 Stage-2 初始化失败
- 修复后验证：最小 two-stage（`1 + 1` iter）可完整结束，日志出现 `[Done] Two-stage training finished.`
