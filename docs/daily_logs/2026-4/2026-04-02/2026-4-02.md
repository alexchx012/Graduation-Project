### 3. 关键发现：Seed 敏感性问题

**Seed42 全部成功**（15/15 policies mean_vx > 0.95），但 **seed43/44 部分退化**：

| 分类 | Policies |
|------|----------|
| 全 seed 可走 (vx > 0.8) | P2 (42,43), P3 (42,43), P10 (42,43) |
| 仅 seed42 可走 | P1 (43,44 ≈ 0), P4 (43,44 ≈ 0) |
| seed44 全部退化 | P1, P2, P3, P4, P10 的 seed44 均 vx < 0.02 |

**seed44 退化模式特征：**
- mean_vx ≈ 0.02（近似站着不动）
- J_smooth ≈ 0.04（极低 = 几乎没有动作变化）
- J_energy 显著偏高（200-335，站立时 actuator 持续耗能但不做功）

**推测原因：** Curriculum 修复（warmup 300 + ramp 300）对 seed42 有效，但 seed44 的随机初始化导致 PPO 在 warmup 期间学到的 scaffold policy 质量不够，进入 ramp 阶段后仍被 MORL secondary rewards 压制。seed43 部分成功说明这是概率性的，与初始参数分布有关。

### 6. Seed 敏感性根因分析

#### 现象

| Seed | A 层可走 (vx > 0.8) | A 层退化 (vx < 0.05) |
|------|---------------------|---------------------|
| 42 | 5/5 (P1,P2,P3,P4,P10) | 0/5 |
| 43 | 3/5 (P2,P3,P10) | 2/5 (P1,P4) |
| 44 | **0/5** | **5/5** |

B 层仅跑了 seed42，5/5 全部成功。

seed44 退化特征：mean_vx ≈ 0.02, J_smooth ≈ 0.04（几乎不动），J_energy 200-335（站立耗能）。

#### 根因：Fresh Adam Optimizer + init_checkpoint

**因果链：**

1. `--init_checkpoint` 当前实现调用 `runner.load(path, load_optimizer=False)` — **故意丢弃 Adam optimizer state**
2. PPO 使用 Adam 优化器。Adam 依赖两个累积量：一阶矩估计（exp_avg，动量）和二阶矩估计（exp_avg_sq，自适应学习率）
3. 新建的 Adam 的 exp_avg=0, exp_avg_sq=0。根据 Adam 公式，bias correction 使**第一步的有效学习率 ≈ 正常的 20 倍**
4. 不同 seed 产生不同的第一个 mini-batch 梯度。seed42 恰好得到有利方向（保持行走），seed44 得到不利方向（破坏行走能力）
5. 这个破坏发生在 curriculum warmup 期间，但 300 iter 的 warmup 不足以让被破坏的 policy 恢复
6. 进入 ramp 阶段后，MORL secondary rewards 进一步放大"站着不动"的吸引力

**验证：** baseline checkpoint `model_1499.pt` 确实**包含** `optimizer_state_dict`（Adam state with 1499 steps of momentum history），但被 `load_optimizer=False` 丢弃。

#### 修复方案

在 `train.py` 新增 `--init_with_optimizer` 标志：
- 与 `--init_checkpoint` 联用时，**同时加载 Adam optimizer state**
- 迭代计数器仍然重置为 0（保证 curriculum 正确计时）
- Adam 继承 baseline 的 1499 步动量估计 → 第一步 update 幅度正常

**改动文件：**

| 文件 | 改动 |
|------|------|
| `scripts/go1-ros2-test/train.py` | 新增 `--init_with_optimizer` CLI 参数；`init_checkpoint` 加载逻辑根据该标志决定是否加载 optimizer state |
| `scripts/phase_morl/run_morl_train_sweep.py` | `_build_train_cmd()` 支持 `exp["init_with_optimizer"]` 字段 |
| `scripts/phase_morl/run_morl_confirm_sweep_v2.py` | `_build_v2_experiment()` 默认启用 `init_with_optimizer=True` |
| `scripts/phase_morl/run_morl_ablation.py` | 同步启用 `init_with_optimizer=True` |

#### 为什么不用其他方案

| 方案 | 原因 |
|------|------|
| 延长 warmup (500+500) | 不治本：第一步仍是 20x 大，只是给更多恢复时间，浪费 GPU |
| Learning rate warmup | 需要修改 rsl_rl 内部或复杂的 scheduler 注入 |
| 完全 resume（不重置 iter） | Curriculum 从 step 1499×24 开始，warmup/ramp 阶段直接被跳过 |
| 固定只用 seed42 | 统计无意义，论文不可接受 |
